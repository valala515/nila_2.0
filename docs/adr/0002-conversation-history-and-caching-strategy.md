# 0002. Таблица разговоров: sessions как first-class сущность, денормализация вместо кэша, keyset-пагинация

## Статус

Принято, 15 июля 2026.

## Контекст

PM попросил в дашборде список последних разговоров пользователя с пагинацией,
фильтром по userId и по тому, до какой фазы дошёл/на какой бросил
пользователь, экспорт подходящих разговоров и просмотр каждого целиком
(вопрос бота + ответ пользователя по очереди).

На момент запроса soft-reset (0001 не покрывал это — реализован позже в той
же сессии) хранил только одну последнюю границу
(`profiles.session_reset_after_turn_id`, ещё не задеплоено). Этого достаточно,
чтобы отделить "текущую" сессию от прошлых ходов в контексте interview engine,
но недостаточно, чтобы перечислить прошлые тестовые заходы как отдельные
строки таблицы — граница перезаписывалась, история границ терялась.

Также реплики бота никогда не сохранялись (`turns` — только пользователь, по
дизайну по `role filter: user only`), что делает невозможным полноценный
транскрипт разговора.

Отдельное явное требование PM: дашборд должен открываться быстро даже когда
данных станет много — тяжёлая таблица разговоров не должна блокировать
основные KPI, и данные не должны пересчитываться заново на каждый рендер.

## Решение

### 1. `sessions` — first-class сущность, а не производная от `turns`

Новая таблица `sessions` (id, user_id, started_at_iso, ended_at_iso,
last_activity_at_iso, reached_phase, completed_at_iso, turn_count,
total_answer_chars). `turns` и новая `bot_messages` получают `session_id`.
`SessionPort.openNewSession` закрывает предыдущую открытую сессию
пользователя и открывает новую — вызывается на `/start` и на soft-reset.
`profiles.session_reset_after_turn_id` (ещё не задеплоенный маркер) заменён
этим решением, а не мигрирован — конверсия данных не нужна.

Альтернатива (отклонена) — оставить маркер одной последней границы: тогда
таблица разговоров первое время видит только текущую сессию каждого
пользователя, прошлые тестовые заходы физически не восстановить (данные не
удалены, но граница между ними стёрта). Несовместимо с прямым запросом PM
("последние 25 разговоров").

### 2. Денормализация агрегатов вместо агрегатных запросов на чтение

`reached_phase`, `turn_count`, `total_answer_chars`, `completed_at_iso` в
`sessions` обновляются один раз на ход (там, где значение и так уже известно
— в `advanceInterview.ts`), а не вычисляются `JOIN`/`GROUP BY` по `turns`/
`events` на каждый рендер таблицы. Чтение списка разговоров — `SELECT` без
агрегации даже когда `turns` вырастет до миллионов строк.

Альтернатива (отклонена сейчас) — TTL-кэш в памяти или periodic rollup-таблица
для тяжёлых запросов. Денормализация закрывает основной риск для этой
конкретной таблицы дешевле и без дополнительной инфраструктуры/инвалидации.

### 3. Keyset-пагинация, не OFFSET

`listConversations` фильтрует по `id < cursor ORDER BY id DESC LIMIT N` —
константное время по индексу PK независимо от глубины пагинации. Плата: нет
прямого перехода на произвольную страницу, только вперёд/назад по цепочке
курсоров (UI хранит стек посещённых курсоров).

Альтернатива (отклонена) — `OFFSET`-пагинация: проще для "перехода на
страницу N", но сканирует и отбрасывает N строк на каждый запрос — дороже с
каждой следующей страницей, ровно то, от чего предостерегал PM ("в будущем
может выдавать 502").

### 4. Экспорт — софт-кап 500 разговоров, без отдельного COUNT-запроса

`GET /api/conversations/export` запрашивает `limit+1` за один SQL-вызов,
чтобы понять, есть ли обрезание (`truncated`), не считая точный `totalMatching`
отдельным агрегатным запросом — та же логика "не пересчитывать лишнего".

## Что заставит пересмотреть

- Если `events`-таблица (используется `/api/dashboard-metrics`, не
  `sessions`) вырастет настолько, что дашборд ощутимо тормозит в проде —
  тогда либо короткий TTL-кэш на `/api/dashboard-metrics`, либо periodic
  rollup-таблица становятся оправданными.
- Если PM попросит переход на произвольную страницу (не только вперёд/назад)
  — потребуется либо гибридная OFFSET-пагинация с ограничением глубины, либо
  показ приблизительного номера страницы поверх keyset-курсоров.
- Если объём разговоров одного экспорта регулярно упирается в софт-кап 500 —
  повод добавить постраничный/потоковый экспорт вместо одного JSON-ответа.
