# 0001. Стек PM-дашборда: кастомный SPA + events-таблица в существующей SQLite

## Статус

Принято, 15 июля 2026.

## Контекст

Продакт-менеджеру нужен дашборд с 6 core-метриками интервью (completion,
felt heard, cost per session и т.д.), drop-off funnel с drill-down по
ходу/вопросу интервью, переключением chart/table и минимальной иконографикой.
До этого решения в проекте не было ни структурированной событийной схемы
(`docs/sprint-plan.md` фиксирует: "Event names — отсутствуют"), ни какого-либо
аналитического/admin-слоя — только Telegram-бот и SQLite (`turns`, `profiles`).

Рассмотрены три варианта:

1. **PostHog (self-host)** — готовые funnels/drop-off и даже session replay из
   коробки. Но: собственный event-store (ClickHouse/Postgres/Redis/Celery),
   не читает существующую SQLite напрямую — события пришлось бы дублировать в
   отдельный SaaS/сервис. PostHog Cloud (без self-host) снимает часть этой
   тяжести, но всё ещё внешний SaaS для health-related данных пользователей.
2. **Metabase** — самый быстрый time-to-first-chart на существующей SQLite,
   chart/table toggle есть из коробки. Но: официальная документация Metabase
   не рекомендует SQLite как серьёзный source в проде (single-writer,
   блокировки), а funnel по adaptive-интервью (turn+field) пришлось бы
   вручную писать на SQL без изменения интерфейса, теряя контроль над формой
   funnel-шага.
3. **Кастомный SPA** (React + Recharts + lucide-react), читающий SQLite
   напрямую через новый read-слой в этом же Node-процессе.

## Решение

Кастомный SPA (`dashboard/`, отдельный Vite-пакет) + новая таблица `events` в
существующей SQLite + маленький HTTP-транспорт (`src/transport/dashboardApi.ts`)
**в том же процессе**, что и Telegram-бот (общий `better-sqlite3`-коннекшн,
не два независимых процесса-писателя/читателя одного WAL-файла).

Обоснование:
- Единица funnel-шага (`turnNumber` + `fieldKey` из одного события
  `interview_turn_answered`) специфична для adaptive-интервью этого продукта —
  готовые funnel-инструменты (PostHog/Metabase) не знают про profile fields
  и потребовали бы либо кастомной модели событий поверх них (PostHog), либо
  ручного SQL без нужного UI (Metabase). Полный контроль важнее скорости
  запуска на этом этапе (1 инженер, 8-недельный MVP).
- SQLite остаётся единственным источником данных проекта — не хотим второй
  БД/сервис только ради дашборда, пока основной стек ещё не мигрировал на
  Postgres (см. открытый вопрос "место в архитектуре" в `docs/sprint-plan.md`).
- Никаких новых внешних SaaS с доступом к health-related данным пользователей
  (соответствует CLAUDE.md §5 — приватность как явный приоритет).

## Альтернативы (отклонены)

- PostHog (self-host или cloud) — см. контекст.
- Metabase — см. контекст.
- Grafana — рассмотрен и отклонён на этапе ресёрча: нет нативного funnel
  primitive для пошаговых user-flow, больше подходит для infra/ops-метрик.

## Что заставит пересмотреть

- Рост объёма событий/пользователей, при котором single-process
  SQLite-writer (бот + dashboardApi в одном процессе) начинает деградировать
  по latency или WAL-файл растёт быстрее, чем приемлемо для дискового I/O —
  тогда возврат к Metabase/Postgres или вынос дашборда в отдельный сервис
  становится обоснованным.
- Решение по миграции основного стека на Postgres (Sprint 5, зависит от
  корректировки №3 в `docs/sprint-plan.md` про место `nila_2.0` в архитектуре)
  — на Postgres Metabase становится заметно более конкурентным вариантом.
- Запрос на функциональность, которую PostHog/Metabase дают "бесплатно"
  (session replay, cohort-анализ, alerting) и которую дорого поддерживать
  самостоятельно в кастомном SPA.
