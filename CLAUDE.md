# Правила работы с этим проектом (Nila AI / The You Who Made It)

Этот файл — baseline для любой работы в этом репозитории, свой или через Claude
Code. Источник правил: «Архитектура, читаемость, анти-раздутие — краткий свод»
(`docs/input/`) + вынесенный опыт `ai_agent_chat` (legacy-проект, где часть этих
правил была прописана только в прозе и не соблюдалась инструментами — не
повторяем эту ошибку). Цель: раздутых файлов/функций 500+ строк, «shallow
modules» и непрослеживаемых зависимостей быть не должно **с первого коммита**,
а не после факта.

---

## 1. Архитектура: слои и направление зависимостей

Проект строится по hexagonal/clean-architecture слоям. Направление зависимостей
— **строго внутрь**, домен ничего не знает об инфраструктуре и транспорте.

```
src/
  domain/            # чистые бизнес-правила, без I/O, без внешних SDK
  application/        # use cases — оркестрация домена, вызывает порты
    ports/             # интерфейсы (LlmPort, SpeechToTextPort, SchedulerPort, ...)
  infrastructure/      # реализации портов: LLM/STT-адаптеры, БД, scheduler, логгер
  transport/           # Telegram-хендлеры — только parse → use case → format reply
  prompts/             # версионированные промпты (никаких inline-шаблонов в коде)
  config/              # env.ts — типизированная и провалидированная конфигурация
docs/
  domain-glossary.md   # он же CONTEXT.md/LANGUAGE.md из свода — единый словарь терминов
  architecture.md
  adr/                 # 0001-title.md — один ADR на спорное/необратимое решение
tests/
  unit/  contract/  integration/  e2e/
```

Правило: `domain/` не импортирует ничего из `infrastructure/` или `transport/`;
`application/` импортирует только `domain/` и `ports/`. Это не гентльменское
соглашение — граница проверяется тестом (`dependency-cruiser`, см. §4).

Соответствие терминов из свода: `entrypoints/` свода = `transport/` здесь,
`data-access/` свода = часть `infrastructure/` здесь. `application/` — слой,
которого в кратком своде нет явно, но без него use case'ы либо утекают в
`transport/`, либо в `domain/`, что нарушает deep-module principle ниже.

## 2. Порядок работы над фичей (Claude Code skills)

Пять skills покрывают разные моменты цикла — вопрос не «какой один», а «что
когда запускать»:

| Skill | Когда | Что делает |
|---|---|---|
| `simplify` | Сразу после фичи, до code-review | Чистит reuse/simplification/efficiency в изменённом коде, пока свежо в контексте. Багов не ищет — только качество. |
| `code-review` | Перед мержем, на весь diff | Correctness-баги + reuse/simplification findings. effort low/medium — быстрая проверка, high/max — для рискованных изменений. |
| `verify` | После нетривиального изменения, перед коммитом | Гоняет фичу end-to-end по-настоящему — ловит «тесты зелёные, фича не работает». |
| `security-review` | Перед деплоем изменений auth/данных/внешних API | Отдельный проход по security. |
| `improve-codebase-architecture` | Раз в N спринтов, или когда что-то ощущается архитектурно «не так» — не на каждый PR | Диагностика по Ousterhout: ищет shallow modules и предлагает deepening. Тест: удаление модуля концентрирует сложность → модуль оправдан; сложность расползается по вызывающим → shallow. |

**Порядок на фичу:** написал код → `simplify` → `code-review` → `verify` →
коммит. `improve-codebase-architecture` — отдельный ритм, не привязан к
конкретному PR.

## 3. Читаемость кода

- **Deep module principle (Ousterhout).** Модуль тем лучше, чем меньше его
  интерфейс относительно того, что он скрывает внутри. Тонкие get/set-обёртки
  без логики — признак плохого модуля; их разумнее заинлайнить, а не выносить
  «ради вынесения».
- **Один уровень абстракции на функцию.** Не смешивать «что делать»
  (`validateUser(); chargeCard(); sendReceipt();`) и «как именно» (конкретный
  SQL, парсинг строки) в одной функции. Смешение уровней — сигнал extract.
- **Guard clauses вместо вложенности.** Ранний `return`/`throw` на невалидный
  случай в начале функции, не оборачивание всего тела в `if (valid) { … }`.
  Каждый уровень вложенности — нагрузка на читателя (см. также hard limit
  nesting depth ниже).
- **Именованные булевы вместо сложных условий:**
  ```ts
  // вместо: if (isEligibleForTrial && !hasActiveSubscription && daysSinceSignup < 14) { … }
  const canStartTrial = isEligibleForTrial && !hasActiveSubscription && daysSinceSignup < 14;
  if (canStartTrial) { … }
  ```
- **Единый словарь терминов.** Если «цель» в коде то `goal`, то `objective`, то
  `target` — это ложные концептуальные различия. `docs/domain-glossary.md`
  заводится с первого дня, новые термины вписываются по мере кристаллизации.
- **Тесты как документация интерфейса.** Если объяснить, что делает функция,
  можно только прочитав её тело, а не сигнатуру и тесты — интерфейс
  недостаточно ясен.
- **SPEC-блоки** для функций со сложной логикой, побочными эффектами или
  инвариантами (перенесено из `ai_agent_chat/CLAUDE.md`, работающая практика):
  ```ts
  /**
   * SPEC: имяФункции
   * Назначение: что делает и зачем существует
   * Входы/Выход: параметры и что возвращает, при каких условиях
   * Разрешённые side effects: только явно перечисленные
   * Инварианты: что функция гарантирует и не должна нарушать
   * Запрещено: что нельзя добавлять без пересмотра SPEC
   */
  ```
  SPEC — контракт: сначала меняется SPEC, потом код.
- **Intent-based именование.** Имя отвечает на «что это?»/«зачем?», не «когда
  добавили?». Запрещены патч-префиксы (`_v2_handler`, `_fix1_result`),
  непрозрачные сокращения, двойное отрицание (`skipIfNotActive` вместо
  `isActive`). Пример: `isGoalActive` не `_hasGoal`, `deriveProgressFromActivity`
  не `_calcPr1`.
- **Word boundary при текстовом поиске.** Всегда `\bword\b`, никогда голое
  `word` — иначе `stress` матчит `stressful`, `yoga` матчит `Yoga That Works`
  (реальный баг в legacy-проекте).
- **"Why"-комментарий** — одна строка над неочевидным местом, только когда без
  неё читатель может удивиться или сломать код при изменении. Отвечает на
  «почему?», не «что делает?» (это уже даёт имя). Если код читается без
  комментария — не писать его.
- **Правило для истории/логов в функциях**: любая функция, сканирующая
  `history`/переписку, обязана в комментарии указать `role filter: "user"
  only` или `role filter: "all"` с обоснованием.
- **Index-time enrichment** для семантического анализа данных: обогащение при
  загрузке (`item._summary`, `item._tags`), а не LLM-вызов в рантайме на каждый
  запрос; graceful degradation обязательна — если обогащение упало, поле
  `null`, откат к базовому поведению без краша.

## 4. Анти-раздутие — с первого коммита, не постфактум

Прошлый монолит (`ai_agent_chat`) вырос до 6000+ строк через десятки отдельных
«безопасных» изменений — лимиты появились только на разборе постфактум. Здесь
пороги настраиваются в скелете проекта до первой строчки бизнес-логики.

**Числовые baseline (hard limit = CI-blocking error, не warning):**

| Метрика | Soft target | Hard limit |
|---|---|---|
| Строк в файле | 250 | 400 |
| Строк в функции | 40 | 80 |
| Cyclomatic/cognitive complexity | — | 10 |
| Глубина вложенности | — | 3 |
| Число параметров | — | 5 |

```js
// eslint.config.js — фрагмент
rules: {
  'max-lines': ['error', { max: 400, skipBlankLines: true, skipComments: true }],
  'max-lines-per-function': ['error', { max: 80, skipBlankLines: true, skipComments: true }],
  'complexity': ['error', 10],
  'max-depth': ['error', 3],
  'max-params': ['error', 5],
  'sonarjs/cognitive-complexity': ['error', 10],
}
```

Почему `error`, а не `warn`: `warn` не останавливает — на него научаются
закрывать глаза за неделю. `error` ломает `npm run lint`, а значит ломает CI.
Именно так рос предыдущий монолит: ни один отдельный PR не выглядел как
раздутие.

1. **Pre-commit hook (husky + lint-staged)** — превышение лимита не попадает в
   git вообще, а не ловится на code review постфактум.
   ```bash
   npx husky add .husky/pre-commit "npx lint-staged"
   # lint-staged: { "*.ts": ["eslint --max-warnings=0"] }
   ```
2. **Тот же линт — обязательный шаг CI**, не только локальный hook. Hook можно
   обойти `--no-verify`, CI — нельзя.
3. **Архитектурный скелет с первого дня** (см. §1) — если с первого дня нет
   единого файла, куда «удобно дописать ещё один if», раздувать физически
   некуда.
4. **Триггер extract по сути, не только по строкам**: как только функция
   начинает делать 4-е по счёту дело (валидация → бизнес-правило → запись в БД
   → форматирование ответа — уже 4) — остановиться и вынести, не дожидаясь
   порога линтера. Линтер — страховка на случай, если триггер пропустили, а не
   основной механизм.
5. **Периодический отчёт по всему проекту, не только по diff**: раз в спринт —
   прогон complexity-репорта (`@genese/complexity` или `escomplex`) по всей
   кодовой базе. Линтер на PR ловит новые нарушения, но не покажет, что файл
   вырос до 380 строк за 15 маленьких «безопасных» PR подряд.
6. **Dependency boundary check** (`dependency-cruiser`): `domain/` не может
   импортировать `infrastructure/`/`transport/`; `application/` — только
   `domain/` и `ports/`. Проверяется в CI, не только на глаз.

## 5. Приватность и логирование

Перенесено из `ai_agent_chat/observabilityService.js` (единственная
безоговорочно хорошая практика legacy-проекта): **никогда не логировать сырой
текст сообщений пользователя, транскрипты голоса или health-контекст** — только
хэши/категории. Полные ответы модели/пользователя не попадают в логи и
Sentry-контекст.

## 6. Промпты

Промпты живут как версионированные файлы в `prompts/` (`interviewQuestionPrompt.v1.md`),
код ссылается на них по версии. Никаких inline-шаблонных строк внутри
TypeScript-файлов — это была системная слабость `systemPromptBuilder.js` в
legacy-проекте (445 строк, промпт и код нераздельны).

## 7. Конфигурация

`config/env.ts` — типизированная и провалидированная через Zod конфигурация,
проверяется при старте процесса; при невалидной конфигурации процесс не
стартует с понятной ошибкой, а не падает на первом обращении к `process.env`.

## 8. Тесты

Четыре уровня: `unit/` (домен, чистые функции), `contract/` (ответы
LLM/STT-адаптеров валидируются против schema-фикстур), `integration/`
(репозиторий + scheduler против реальной БД), `e2e/` (сценарий в реальном
Telegram). **Каждый тест использует изолированную БД/scratch-состояние**, не
общий on-disk файл — в legacy-проекте (`ai_agent_chat/tests/schema-contract.test.js`)
тест открывает on-disk `assistant-beta.sqlite` напрямую вместо изолированной
копии; при параллельном запуске тестов это реальный источник flakiness. Это
не повторяем.

## 9. ADR и glossary

Спорное или трудно обратимое решение (например, выбор БД, TypeScript vs JS,
webhook vs long-polling) — отдельный `docs/adr/000N-title.md`: контекст,
решение, альтернативы, что заставит пересмотреть. `docs/domain-glossary.md`
ведётся с первого дня, а не заводится «когда накопится путаница».

## 10. Git-конвенции и версионирование

Формат коммит-сообщений перенесён из `ai_agent_chat` (там он соблюдался в
прозе, здесь — проверяется инструментом, см. §4 про `warn` vs `error`).
Совпадает с глобальными skills `commit`/`git-message`.

**Заголовок:**
```
<тип>(<область>): <краткое описание на русском, до 100 символов>
```
`<тип>` — один из `feat`, `fix`, `refactor`, `style`, `chore`, `docs`, `test`.
Breaking change — `!` после типа/области (`feat(auth)!: ...`) или футер
`BREAKING CHANGE: ...`.

**Тело** (что применимо, не все секции обязательны):
```
Что сделано
- ...

На что повлияет
- ...

Компромиссы
- ...

Метрики
- ...

До / После
До:  ...
После: ...
```

Проверяется `commitlint` (`commitlint.config.cjs`) через husky
`.husky/commit-msg` — некорректный заголовок не даёт закоммитить локально.
Локальный hook обходится `--no-verify` (как и pre-commit, см. §4) — настоящая
проверка происходит в CI: `semantic-release` разбирает заголовки коммитов, и
некорректный/нестандартный тип коммита просто не попадёт в анализ релиза.

**Версионирование — SemVer, авто-бамп в CI.** `.releaserc.json` +
`.github/workflows/release.yml`: при каждом пуше в `main` `semantic-release`
анализирует коммиты с прошлого релиза, поднимает версию в `package.json`
(`feat` → minor, `fix` → patch, breaking → major; `refactor`/`style`/`chore`/
`docs`/`test` сами по себе релиз не создают — иначе версия росла бы шумно на
каждый косметический коммит), генерирует `CHANGELOG.md`, ставит git-тег
`vX.Y.Z` и создаёт **GitHub Release** — это и есть учёт прод-релизов: вкладка
Releases в GitHub — единственный источник правды о том, что когда-либо было
выпущено, без ручного ведения списка версий.

**Трейсинг каждого коммита, а не только релизов.** `scripts/generateVersion.mjs`
запускается перед `build`/`dev` (`prebuild`/`predev`) и пишет
`src/config/version.ts` (в `.gitignore`, генерируется, не коммитится) —
`BUILD_VERSION = "<semver>+<число коммитов>.<короткий sha>"`. Логируется при
старте процесса (`src/index.ts`). Это отдельная ось от SemVer-релиза: даже
коммит, который не поднял версию (например `docs`), получает свой уникальный
и монотонно растущий номер сборки — можно точно сказать, какой коммит сейчас
запущен в проде, не дожидаясь следующего релиза. Когда появится Sentry (или
аналог) — `BUILD_VERSION` должен стать значением поля `release`, чтобы ошибки
сопоставлялись с точным деплоем; пока это не сделано, т.к. Sentry ещё не
подключён в стек.

Важно: в `ai_agent_chat` версия `package.json` (`0.1.0`) никогда не менялась,
тегов нет вообще — версионирование там отсутствовало как практика. Это не
перенос опыта legacy-проекта, а сознательное улучшение поверх него.
